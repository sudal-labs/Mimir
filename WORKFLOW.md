# WORKFLOW.md — 작업 방식 원칙

이 문서는 Claude가 "어떻게 작업할지"에 대한 기본 정책입니다.
구체적인 예외는 OVERVIEW.md 5번 섹션을 참고하세요.

## 응답/커뮤니케이션
- 응답은 한국어로 작성 (OVERVIEW.md에 다른 지정이 있으면 그것을 따름)

## 의사결정 사이클 (제안 → 검수 → 피드백 → 확정)
구조 선택, 데이터 모델링, 스택 도입처럼 트레이드오프가 있는 판단은 한 번에 끝내지 않고
아래 4단계를 거칩니다. 사용자가 확정하기 전까지는 같은 항목을 계속 다듬습니다.

| 상태 표기 | 의미 |
|---|---|
| 🟡 제안 | Claude가 근거/트레이드오프를 정리해 제안한 상태. 아직 확정 아님 |
| 🟢 확정 | 사용자가 최종 승인한 상태 |
| 🔴 기각 | 검토 후 채택하지 않기로 한 상태 (이유와 함께 남김 — 같은 논의가 반복되지 않도록 지우지 않음) |
| ⚪ 대체됨 | 이미 확정됐던 항목이 새 결정으로 교체된 상태 (새 항목으로 링크) |

1. **제안 (Claude)**: 판단 근거 → 트레이드오프 → 이 설계가 깨지는 시점 순서로 작성하고 `🟡 제안`으로 표시
2. **검수 (사용자)**: 제안을 검토하고 피드백/수정 요청/승인 의사를 남김
3. **피드백 반영 (Claude)**: 피드백을 그대로 수용하지 않고, 기존 판단 근거와 충돌하면 트레이드오프를
   다시 짚어 대응합니다. 받아들인 부분은 제안을 수정하고, 기각한 부분은 이유를 남기되 `🟡 제안` 상태를 유지
4. **확정 (사용자)**: 사용자가 최종 확인하면 `🟢 확정`으로 바꾸고 날짜를 기록
   - 나중에 뒤집어야 하면 기존 항목을 지우지 않고 `⚪ 대체됨`으로 바꾸고, 새 항목을 새로 추가

기록 위치는 OVERVIEW.md "작성해야 할 문서"에 지정된 설계 노트를 사용하고, 지정된 곳이 없으면
어디에 남길지 먼저 묻습니다. 이 상태 표기는 STACK.md "확장 스택" 표, AI 협업 로그 등
프로젝트 전체에서 동일하게 사용합니다.

## AI 협업 로그
- 주요 결정에 영향을 준 대화는 전체 로그가 아니라, 결정 근거가 드러나는 핵심 질문/답변만
  짧게 발췌해 남깁니다.
- 위 "의사결정 사이클"의 상태 표기를 그대로 사용해 현재 어느 단계까지 진행됐는지 표시합니다.

## 테스트 정책
- 코드 작성 시 `pytest` 기반 단위 테스트를 항상 함께 추가합니다.
  (OVERVIEW.md 테스트 도구가 다르게 지정되어 있으면 그 도구를 사용합니다)
- 기존 코드를 변경할 때는 관련 단위 테스트도 함께 수정합니다.

## 변경 작업 정책
- 변수명과 함수명은 항상 풀네임을 사용합니다 (축약어 지양).
- 기존 동작/정책(특히 데이터 처리 방식)을 바꿔야 하는 상황이 생기면, 바로 고치지 말고
  트레이드오프를 먼저 제시하고 확인을 받습니다.

## 커밋/PR
커밋 메시지는 아래 형식을 따릅니다.

```
[type](scope): subject

body

footer
```

- **type**: 변경의 성격 (예: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `style`, `perf`)
- **scope** (선택): 변경 영역 (예: `api`, `etl`, `db`)
- **subject**: 한 줄 요약. 한국어, 동사형으로 작성하고 끝에 마침표를 붙이지 않음
- **body** (선택): 왜/무엇을 바꿨는지 설명
- **footer** (선택): 관련 이슈나 설계 노트 참조 (예: `설계노트 #12 참고`, `Closes #123`)

예시:
```
[feat](api): 단가 이력 조회 기능 개발

기존 단가뿐 아니라 시점별 변동 이력을 조회할 수 있도록 추가
설계노트 #12 참고
```
