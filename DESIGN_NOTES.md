## sentence-transformers (로컬 임베딩)
- **선택**: 노트 청크 임베딩을 외부 API 대신 자체 호스팅 모델(sentence-transformers)로 생성
- **왜 이 선택을 했는가**: 외부 API 호출 비용·네트워크 지연 없이 배치로 대량 처리 가능. 노트 내용이 외부로 나가지 않아 개인 vault의 프라이버시도 지킬 수 있음
- **트레이드오프**: 상용 API 임베딩보다 검색 품질이 다소 낮을 수 있음. GPU 없는 ARM 환경(Hetzner CAX31)에서는 CPU 추론이라 처리량이 제한됨
- **설계가 깨지는 시점**: 노트 수가 수만 건 이상으로 늘어나 야간 배치 인덱싱이 허용 시간(1~2시간) 안에 끝나지 않을 때. 이 시점엔 GPU 인스턴스로 임시 스케일업(시간당 과금)하거나, 임베딩을 외부 API로 전환 검토

## GitPython
- **선택**: git 조작(diff 감지, backlink commit, push)을 GitPython으로 자동화
- **왜**: subprocess로 git CLI를 직접 호출하는 것보다 에러 핸들링과 결과 파싱이 구조화되어 있어 vault-ingest/vault-link 구현이 단순해짐
- **트레이드오프**: 순수 CLI 호출보다 약간의 추상화 오버헤드. 일부 git 고급 기능은 결국 CLI를 우회 호출해야 할 수 있음
- **설계가 깨지는 시점**: vault가 매우 커져(수천 개 파일) 매 webhook마다 풀 diff 계산이 느려질 때. 이 시점엔 shallow fetch나 커밋 해시 기준으로 diff 범위를 좁히는 최적화가 필요

## python-frontmatter
- **선택**: 마크다운 frontmatter(YAML 메타데이터) 파싱/작성에 표준 라이브러리 사용
- **왜**: YAML 파싱과 본문 분리를 직접 구현하면 엣지케이스(중첩 구조, 날짜 타입 등)가 많아 표준 라이브러리가 안전
- **트레이드오프**: 라이브러리 의존성 하나 추가. frontmatter 스키마가 단순 key-value를 넘어서는 관계형 데이터를 표현하기엔 부적합
- **설계가 깨지는 시점**: 노트 간 관계를 frontmatter 안에 직접 표현해야 할 정도로 메타데이터가 복잡해질 때. 이미 메타데이터는 PostgreSQL에도 저장하므로, 그 시점엔 frontmatter를 "사람이 읽는 원본 표기"용으로만 한정하고 관계 데이터는 PostgreSQL 쪽으로 완전히 이전

## Alembic
- **선택**: PostgreSQL 스키마 마이그레이션 관리에 Alembic 사용
- **왜**: 노트/링크 그래프 스키마가 초기 구현 단계에서 자주 바뀔 가능성이 높음. 수동 ALTER TABLE보다 버전 관리되는 마이그레이션이 안전하고, 나중에 어떤 이유로 스키마가 바뀌었는지 추적 가능
- **트레이드오프**: 초기 설정 비용(migration 환경 구성)이 있고, 개인 프로젝트 규모치고는 약간 무거울 수 있음
- **설계가 깨지는 시점**: 사실상 거의 없음 — 스키마가 안정화되면 마이그레이션 빈도가 줄어 오버헤드만 남지만, 히스토리 추적 가치가 더 크기 때문에 그래도 유지하는 쪽을 권장

## mcp (Python SDK)
- **선택**: vault-api를 감싸는 MCP 서버를 공식 mcp Python SDK로 구현
- **왜**: MCP 프로토콜(JSON-RPC 기반 stdio/HTTP)을 직접 구현하는 것보다 공식 SDK가 스펙 변경에 안전하고, Claude Code의 `claude mcp add` 커맨드와 바로 호환됨
- **트레이드오프**: SDK 버전 업데이트에 따라 인터페이스가 바뀔 수 있어 추적이 필요함
- **설계가 깨지는 시점**: MCP 툴 호출의 응답 모델로는 부족한 실시간성(예: 매우 긴 답변의 점진적 스트리밍)이 필요해질 때. 이 시점엔 vault-api를 직접 HTTP로 노출해 별도 클라이언트에서 스트리밍 받는 방식을 병행 검토
