## API 컨벤션
- 엔드포인트 경로는 표준 REST 관례에 따라 복수형 명사 사용 (예: `/items`, `/companies/{id}`)
- 응답 본문은 JSON, 필드명은 `camelCase`
- 모든 시각(timestamp)은 한국 시간(Asia/Seoul)으로 저장/응답
- 성공 응답도 실패 응답도 HTTP 200으로 내려가며, 본문의 `status` 필드로 성공/실패 구분
- 새 엔드포인트를 추가할 때도 위 패턴(복수형, camelCase)을 일관되게 따를 것

## DB 컨벤션
- 모든 테이블/컬렉션에 `created_at` / `updated_at` 필드를 추가
- 외부 API 원본 필드명을 그대로 쓰지 않고 내부 도메인 언어(camelCase, 풀네임)로 번역. 외부 식별자 값 자체는 보존
- 외부 소스가 바뀌어도 매핑 레이어만 교체하면 되도록, 내부 필드명과 외부 API 필드명을 분리해서 설계
- 신뢰할 수 없는 외부 참조 키에는 제약을 강제하지 않음 (데이터 보존 우선, 무결성은 별도 품질 체크로 관리)

## 코드 레이어 규칙
- `repository`는 쿼리/저장소 접근만 캡슐화하고 비즈니스 로직을 넣지 않음
- `services`가 비즈니스 로직을 담당하고, repository 결과를 schema(Pydantic)로 가공해 라우터에 전달
- DB 모델(`models`)과 API DTO(`schemas`)는 항상 독립적으로 유지 — 서로 직접 노출하지 않음
- 다른 값들로부터 계산 가능한 파생값은 저장하지 않고, 서비스 레이어(Python)에서 계산

## 주석/네이밍
- PEP 8 네이밍 (함수·변수: `camelCase`, 클래스: `PascalCase`, 상수: `UPPER_SNAKE_CASE`)
- 가능한 모든 함수/메서드 시그니처에 타입힌트 작성
- 공개 함수/클래스에는 책임을 한두 줄로 요약하는 docstring 작성
